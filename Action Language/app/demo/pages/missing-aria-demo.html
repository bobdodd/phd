Detecting Broken ARIA Relationships

The Problem: Broken ARIA Relationships

ARIA (Accessible Rich Internet Applications) uses special attributes to create semantic relationships between elements. Attributes like aria-labelledby, aria-describedby, aria-controls, and aria-owns reference other elements by ID.

When these referenced IDs don't exist, assistive technologies can't establish the relationship, breaking accessibility. Paradise's MissingAriaConnectionAnalyzer validates that all ARIA references point to actual elements.

Example: Broken Label Relationship

A button references a label element that doesn't exist, breaking screen reader announcements.

Broken ARIA Implementation
<!-- The button references 'submitLabel' -->
<button aria-labelledby="submitLabel">
  Submit
</button>

<!-- But no element with id="submitLabel" exists! -->
<!-- Expected: <span id="submitLabel">Submit the form</span> -->

üíî Broken Relationship

Screen reader hears: "Submit, button" (generic, unhelpful)
Should hear: "Submit the form, button" (descriptive, helpful)

The aria-labelledby attribute creates no connection because the target ID doesn't exist. Screen reader users get minimal context about the button's purpose.

ARIA Relationship Attributes

Paradise validates all ARIA attributes that reference other elements:

Attribute Purpose Example Impact if Broken
aria-labelledby Identifies labeling element(s) aria-labelledby="title" Screen readers can't announce proper label
aria-describedby Identifies describing element(s) aria-describedby="help" Users miss important context/instructions
aria-controls Identifies controlled element(s) aria-controls="menu" Relationship between trigger and content lost
aria-owns Defines parent-child relationship aria-owns="item1 item2" DOM structure misrepresented to AT
aria-activedescendant Identifies active element in composite aria-activedescendant="opt1" Focus management broken for keyboard users
aria-flowto Defines reading order override aria-flowto="next" Custom navigation paths don't work
aria-errormessage Identifies error message element aria-errormessage="error1" Users don't hear validation errors

Step 3: Suggest Fixes

Paradise analyzes the context to suggest likely fixes:

  • Missing elements: Suggest creating the referenced element
  • Wrong attribute: Check if relationship should use different ARIA attribute

WCAG 2.1 Compliance

Broken ARIA connections violate multiple WCAG criteria:

  • 1.3.1 Info and Relationships (Level A): Information, structure, and relationships must be programmatically determinable. Broken ARIA breaks this.
  • 4.1.2 Name, Role, Value (Level A): All UI components must have correct name and role. Missing labels break this requirement.
  • 2.4.6 Headings and Labels (Level AA): Labels must describe purpose. Broken aria-labelledby prevents this.
  • 3.3.2 Labels or Instructions (Level A): Labels must be provided when input is required. Broken aria-describedby breaks instructional context.
