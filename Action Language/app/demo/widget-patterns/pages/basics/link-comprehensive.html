Link Pattern

Accessible navigation and hyperlinks

Low Complexity WCAG 2.1.1, 2.4.4, 2.4.9, 4.1.2

Complete Link Pattern Requires:

  • <a> element (not div or span with onclick)
  • href attribute with valid destination
  • Descriptive link text: Text that makes sense out of context
  • Avoid "click here": Link text should describe the destination
  • aria-label for icon-only links (no visible text)
  • aria-describedby for additional context when needed
  • target="_blank" should warn user (opens in new window/tab)
  • role="link" only if NOT using <a> element
  • Keyboard focusable: Native for <a> with href
  • Visual indicators: Underline, color change, or other clear indication
  • Focus visible: Clear focus indicator for keyboard navigation

Why This Fails

  • Fake links (div/span with onclick): Not keyboard accessible, no semantic meaning, screen readers don't announce as links
  • "Click here" links: Meaningless out of context, screen reader users navigating by links can't understand destination
  • Icon-only without label: Screen readers announce "link" but no purpose or destination
  • No href attribute: Link is not keyboard accessible (can't Tab to it)
  • Unexpected new windows: Disorients users, especially screen reader users who may not realize context changed

Code Walkthrough

Basic Link (Accessible)

<a href="/privacy-policy">Privacy Policy</a>

<a> element with href creates a semantic link. Descriptive text makes purpose clear out of context. Naturally keyboard accessible (Tab to focus, Enter to activate).

External Link (Opens New Window)

<a href="https://example.com"
   target="_blank"
   rel="noopener noreferrer"
   aria-label="Example website (opens in new window)">
  Example Website
</a>

target="_blank": Opens in new window/tab
rel="noopener noreferrer": Security best practice
aria-label: Warns users about new window
Alternative: Add visual indicator (icon) and text "(opens in new window)"

Icon-Only Link (Requires aria-label)

<a href="/settings" aria-label="Settings">
  <svg>...</svg>
</a>

aria-label is REQUIRED when link has no visible text. Screen readers announce: "Settings, link" (not just "link"). Alternative: Include visually-hidden text inside link.

Icon Link with Text (aria-label Optional)

<a href="/profile" class="icon-link">
  <svg>...</svg>
  <span>Profile</span>
</a>

When link has visible text, aria-label is NOT needed. Screen readers will announce the visible text: "Profile, link".

Link with Additional Context

<a href="/article" aria-describedby="date">
  Understanding WCAG 2.2
</a>
<span id="date">Published Jan 15, 2026</span>

aria-describedby associates additional context with the link. Screen readers announce: "Understanding WCAG 2.2, link, Published January 15, 2026". Main link text remains concise while providing extra detail.

What NOT to Do

<!-- ❌ BAD: Fake link with span -->
<span onclick="navigate()">Click here</span>

<!-- ❌ BAD: Non-descriptive text -->
<a href="/report">Click here</a>

<!-- ❌ BAD: Link without href -->
<a onclick="doSomething()">Action</a>

<!-- ❌ BAD: Icon-only without label -->
<a href="/settings"><svg>...</svg></a>

<!-- ✅ GOOD: Proper link with descriptive text -->
<a href="/report">Download Annual Report</a>

<!-- ✅ GOOD: Icon-only with aria-label -->
<a href="/settings" aria-label="Settings">
  <svg>...</svg>
</a>

What Paradise Detects

Complete Pattern Validation

Paradise validates semantic link structure and accessibility:

Semantic Elements

  • <a> element (not div/span)
  • href attribute present and valid
  • No fake links with onclick only
  • Proper role (implicit in <a>)

Descriptive Text

  • No "click here" or "read more"
  • Link text describes destination
  • Text meaningful out of context
  • Adequate link text length

Icon Links

  • Icon-only links have aria-label
  • aria-label is descriptive
  • Links with text don't need aria-label
  • No empty links

External Links

  • target="_blank" has warning
  • rel="noopener noreferrer" present
  • User notified about new window
  • Security attributes included

Keyboard Access

  • Links are keyboard focusable
  • Tab order is logical
  • Enter activates link
  • Focus indicator visible

Visual Indicators

  • Links visually distinguishable
  • Underline or other indicator
  • Color not only indicator (WCAG 1.4.1)
  • Focus indicator has 3:1 contrast

WCAG Success Criteria

  • 2.1.1 Keyboard (Level A)

    All link functionality must be available using a keyboard. Links must use <a> with href (naturally keyboard accessible) or have tabindex="0" and Enter key handler.

  • 2.4.4 Link Purpose - In Context (Level A)

    The purpose of each link can be determined from the link text alone or from the link text together with its programmatically determined link context.

  • 2.4.9 Link Purpose - Link Only (Level AAA)

    The purpose of each link can be identified from link text alone. Avoid "click here", "read more", etc. Link text should be descriptive.

  • 4.1.2 Name, Role, Value (Level A)

    Links must have an accessible name (text content or aria-label). Icon-only links MUST have aria-label or visually-hidden text.

  • 3.2.5 Change on Request (Level AAA)

    Opening new windows/tabs should only occur on user request. When using target="_blank", users should be warned (via text or aria-label).

  • 1.4.1 Use of Color (Level A)

    Links must not rely on color alone. Use underline, icon, or other visual indicator in addition to color.
