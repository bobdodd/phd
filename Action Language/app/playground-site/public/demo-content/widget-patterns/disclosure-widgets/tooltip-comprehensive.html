About Tooltips

Tooltips provide supplementary information about an element when users hover over or focus on it. They are commonly used for help text, definitions, and additional context without cluttering the interface.

Accessibility Requirements:

  • role="tooltip" on the tooltip element
  • aria-describedby on trigger element pointing to tooltip ID
  • Show on both hover and focus events
  • Hide on blur and Escape key press
  • Tooltip appears near trigger element
  • Keyboard focusable triggers
  • Escape key dismisses tooltip

Testing Instructions

Keyboard Testing

1. Tab Navigation: Press Tab to navigate to tooltip triggers. In the accessible version, tooltips should appear on focus. In the inaccessible version, they won't.

2. Escape Key: When a tooltip is visible, press Escape. The accessible version should hide the tooltip; the inaccessible version won't respond.

3. Focus Management: Tab away from a trigger with a visible tooltip. The accessible version should hide it automatically.

Screen Reader Testing

Accessible Version: When you focus on a trigger, the screen reader should announce the button label followed by the tooltip description (via aria-describedby).

Inaccessible Version: Screen readers will only announce the button label. Tooltip content remains hidden and inaccessible.

Test with: NVDA (Windows), JAWS (Windows), VoiceOver (macOS/iOS), TalkBack (Android)

Mouse Testing

Both implementations should work with mouse hover. However, only the accessible version provides equivalent keyboard access, which is required for WCAG compliance.

Best Practices

Essential Requirements for Accessible Tooltips:

  • Always use role="tooltip" on the tooltip element
  • Connect trigger to tooltip using aria-describedby
  • Ensure triggers are keyboard focusable (button elements or tabindex="0")
  • Show tooltip on both mouseenter and focus events
  • Hide tooltip on both mouseleave and blur events
  • Implement Escape key to dismiss visible tooltips
  • Position tooltips near their triggers
  • Keep tooltip content concise and relevant
  • Don't hide essential information in tooltips
  • Use sufficient color contrast (4.5:1 for text)

Common Mistakes to Avoid:

  • Using title attribute alone (inconsistent screen reader support)
  • Making tooltips only appear on hover
  • Forgetting to make tooltip triggers focusable
  • Not providing Escape key functionality
  • Putting critical information only in tooltips
  • Using tooltips on non-interactive elements without making them focusable
  • Not connecting tooltip to trigger with aria-describedby
  • Missing role="tooltip" attribute

WCAG 2.1 Criteria

1.3.1 Info and Relationships (Level A)

Information, structure, and relationships conveyed through presentation must be programmatically determined. The accessible implementation uses role="tooltip" and aria-describedby to establish proper relationships.

2.1.1 Keyboard (Level A)

All functionality must be available from a keyboard. The accessible implementation allows keyboard users to trigger and dismiss tooltips using focus and Escape key.

4.1.2 Name, Role, Value (Level A)

For all user interface components, the name and role must be programmatically determined. The accessible implementation provides proper role and associated descriptions.

2.1.2 No Keyboard Trap (Level A)

Keyboard focus can be moved away from a component using only a keyboard interface. The accessible implementation allows users to tab away and use Escape to dismiss.
